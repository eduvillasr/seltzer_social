# Shipping Seltzer Social to the App Store & Google Play

The web app is wrapped in a native shell with **Capacitor**. The shell loads
the deployed site (`server.url` in `capacitor.config.ts`) inside a native
WebView, so you keep one codebase and every web deploy updates both apps
instantly — no resubmission needed for content changes.

> You need a **Mac + Xcode** for iOS and **Android Studio** for Android.
> iOS apps cannot be built on Windows. Do the iOS steps on a Mac.

---

## 1 · Install dependencies

```bash
npm install
```

This pulls in `@capacitor/core`, `@capacitor/cli`, the iOS/Android platforms,
and the splash-screen / status-bar plugins (already added to `package.json`).

## 2 · Point the shell at your production URL

Edit `capacitor.config.ts` → `server.url` and set it to your deployed HTTPS
domain (e.g. `https://seltzer-social.vercel.app`). The native app loads this
URL on launch.

> Tip: for local testing against `next dev`, you can temporarily set
> `server.url` to your machine's LAN IP (`http://192.168.x.x:3000`) and add
> `cleartext: true`. Revert before a release build.

## 3 · Add the native platforms

```bash
npx cap add ios       # run on a Mac
npx cap add android
npx cap sync
```

This creates `ios/` and `android/` native projects (git-ignored is fine; they
regenerate from config).

## 4 · Generate icons & splash screens

The app's icon is generated by the web route `/app-icon`. Grab a 1024px master
and let `@capacitor/assets` produce every required size:

1. Run the app (`npm run dev`) and open
   `http://localhost:3000/app-icon?size=1024` → save the PNG as
   `assets/icon.png` (create the `assets/` folder).
2. (Optional) Save `http://localhost:3000/app-icon?size=1024&maskable=1` as
   `assets/icon-foreground.png` for a padded Android adaptive foreground.
3. Generate:

   ```bash
   npm run cap:assets
   npx cap sync
   ```

This writes all iOS app-icon sizes, Android mipmaps (incl. adaptive +
maskable), and splash screens using the brand colors (`#0891b2` icon bg,
`#0a0e1a` splash bg).

## 5 · Open & run the native projects

```bash
npm run cap:ios       # opens Xcode
npm run cap:android   # opens Android Studio
```

- **iOS:** set your Team / signing in Xcode → Signing & Capabilities, pick a
  device/simulator, press Run.
- **Android:** let Gradle sync, pick a device/emulator, press Run.

## 6 · Auth redirect allow-list (important)

Supabase email-confirmation / OAuth callbacks must allow the in-app origin.
In Supabase → Authentication → URL Configuration → Redirect URLs, add your
production callback (already there if web works) — the WebView uses the same
`server.url` origin, so no extra entry is usually needed. If you switch to
deep-link auth later, add `app.seltzersocial://` style URLs here.

## 7 · Store submission checklist

**Google Play**
- [ ] Build a signed **AAB**: Android Studio → Build → Generate Signed Bundle.
- [ ] Create the app in Play Console, upload the AAB to internal testing first.
- [ ] Listing assets: 512×512 icon (use `/app-icon?size=512`), feature graphic
      1024×500, and **2–8 phone screenshots** (capture from a device/emulator).
- [ ] Fill out the Data Safety form (you collect email + user content via
      Supabase) and the content rating questionnaire.

**Apple App Store**
- [ ] Archive in Xcode → Product → Archive → Distribute to App Store Connect.
- [ ] App Store Connect: 1024×1024 marketing icon (use `/app-icon?size=1024`),
      screenshots for 6.7" and 6.5"/5.5" sizes.
- [ ] Privacy nutrition labels (email + user-generated content).
- [ ] ⚠️ Apple may reject a thin WebView wrapper under guideline 4.2 if it
      feels like "just a website." Mitigations already in place: standalone
      display, native splash/status bar, offline fallback, install-grade
      icons. If reviewers push back, lean on native plugins (push
      notifications, camera for review photos, share sheet) to add native
      value.

## 8 · Native push notifications (likes, etc.)

Like-notifications already write a row to the Supabase `notifications` table.
Native push fans those rows out to devices over **FCM** (FCM relays to APNs for
iOS, so the server only talks to one API).

**Pieces already in the codebase:**
- `supabase_device_tokens.sql` — `device_tokens` table (run it in the SQL editor).
- `lib/push.ts` + `components/PushRegistrar.tsx` — registers the device & stores
  its FCM token after sign-in; routes notification taps to the deep link.
  Mounted in `app/layout.tsx`. No-op on the web.
- `supabase/functions/push-fanout/index.ts` — Edge Function that sends a push
  for every new `notifications` row.

**One-time setup:**

1. **Firebase project** → add an Android app (package `app.seltzersocial`) and an
   iOS app (bundle id `app.seltzersocial`). Download `google-services.json`
   (Android → `android/app/`) and `GoogleService-Info.plist` (iOS → add in Xcode).
2. **iOS only:** in the Apple Developer portal create an **APNs auth key (.p8)**
   and upload it to Firebase → Project Settings → Cloud Messaging. Enable the
   **Push Notifications** capability in Xcode → Signing & Capabilities.
3. Install the plugin (already in `package.json`) and sync:
   ```bash
   npm install
   npx cap sync
   ```
4. **Deploy the Edge Function** and set its secrets:
   ```bash
   npx supabase functions deploy push-fanout --no-verify-jwt
   npx supabase secrets set \
     FIREBASE_PROJECT_ID=your-project-id \
     FIREBASE_CLIENT_EMAIL=firebase-adminsdk-...@your-project.iam.gserviceaccount.com \
     FIREBASE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n"
   ```
   (`SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are injected automatically.)
   Get the service-account values from Firebase → Project Settings →
   Service Accounts → Generate new private key.
5. **Wire the trigger:** Supabase Dashboard → Database → Webhooks → *Create* →
   table `public.notifications`, event **Insert**, type **HTTP Request**, URL =
   the deployed `push-fanout` function URL. (Add an `Authorization` header if you
   deployed without `--no-verify-jwt`.)

After that, any new `notifications` row (likes, comments, mentions, tier invites)
delivers a native push to the recipient's devices. Dead tokens are pruned
automatically when FCM reports them unregistered.

## After config changes

Run `npx cap sync` whenever you edit `capacitor.config.ts` or add a plugin.
Content/UI changes ship automatically via your web deploy — no rebuild needed.
